# 전역 원칙

## 보안 최우선

모든 작업에서 보안을 최우선으로 설계한다.

- 해킹 및 침해 위협에 취약한 구조는 절대 사용하지 않는다
- 보안 관련 결정은 2중 3중으로 검토 후 진행한다
- OWASP Top 10 기준으로 취약점을 항상 점검한다
- 인증/인가, 입력값 검증, 암호화는 타협하지 않는다
- 의심스러운 설계가 있으면 구현 전에 반드시 경고한다

## 금융 정보 보호

금융 및 결제 관련 정보는 최고 수준으로 보호한다.

- 결제 정보, 카드번호, 계좌번호는 절대 평문 저장하지 않는다
- 금융 데이터는 암호화 저장, 최소 권한 접근 원칙을 따른다
- 이상 결제, 정보 탈취 가능성이 있는 설계는 즉시 중단하고 대안을 제시한다
- PCI DSS 기준을 참고하여 결제 흐름을 설계한다

## GitHub 및 코드 관리

- API 키, 비밀번호, 토큰 등 민감 정보는 절대 코드에 하드코딩하지 않는다
- .env 파일은 절대 GitHub에 올리지 않는다 (.gitignore 필수 확인)
- 시크릿은 환경변수 또는 별도 시크릿 매니저로 관리한다

## 코딩 원칙 (Karpathy 4원칙)

### 1. 코딩 전에 먼저 생각하라
- 추측하지 않는다. 모호하면 반드시 질문한다
- 여러 해석이 가능하면 모든 옵션과 장단점을 제시한다
- 자신이 세운 가정을 명시적으로 밝힌다

### 2. 단순함을 최우선으로
- 문제를 해결하는 최소한의 코드만 작성한다
- 요청하지 않은 기능, 추상화, 확장성을 추가하지 않는다
- 한 번 쓸 코드에 디자인 패턴을 적용하지 않는다

### 3. 수술적으로 필요한 부분만 수정
- 요청과 직접 관련된 코드만 수정한다
- 인접한 코드를 임의로 "개선"하지 않는다
- 정상 작동하는 코드를 리팩토링하지 않는다
- 기존 스타일과 컨벤션을 그대로 유지한다
- 불필요해 보이는 코드는 삭제하지 말고 언급만 한다

### 4. 목표 기반으로 실행
- 검증 가능한 성공 기준을 먼저 정의한다
- 버그 수정 시: 버그를 재현하는 테스트를 먼저 작성하고, 그 테스트를 통과시킨다
- 각 단계마다 성공 여부를 검증하며 반복한다
